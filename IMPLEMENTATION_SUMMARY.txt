================================================================================
KEY FUNCTIONS
================================================================================

store.py::claim_next_job(worker_id: str = "default") -> Optional[dict]
  - Atomically claims oldest queued job
  - Tracks worker_id, claimed_at timestamp
  - Returns updated job dict with new status='running'

store.py::detect_orphaned_jobs(stale_after_minutes: int = 0) -> list[dict]
  - Finds jobs in 'running' state with claimed_at > threshold
  - Uses DEEP_THINK_ORPHAN_TIMEOUT_MINUTES env var (default 5)
  - Returns list of orphaned job dicts

store.py::requeue_orphaned_job(job_id: str, reason: str) -> bool
  - Resets job status to 'pending'
  - Clears claimed_by, claimed_at, started_at
  - Returns True if successful, False otherwise

worker.py::_orphan_watchdog(check_interval_seconds: int = 30) -> None
  - Async coroutine for background job monitoring
  - Runs continuously, checking every check_interval_seconds
  - Calls detect_orphaned_jobs() and requeues orphans
  - Updates metrics on detection and requeue

metrics.py::increment_orphaned_jobs_detected()
  - Thread-safe counter increment for detected orphans

metrics.py::increment_orphaned_jobs_requeued()
  - Thread-safe counter increment for requeued orphans

PERFORMANCE CHARACTERISTICS
================================================================================

Memory:
  - Watchdog: negligible (~1KB per orphan check)
  - Job tracking: 2 additional TEXT fields per job (~32 bytes)
  - Overall impact: < 1% for typical workloads

CPU:
  - Watchdog: 0.1% average (sleeps 99% of time)
  - Detection: O(n) where n = number of running jobs
  - Requeue: O(1) per job

Database:
  - Queries: 1 SELECT + 1 UPDATE per orphan detected
  - Index: claimed_at query naturally filtered by status='running'
  - Throughput: < 100 queries/hour typical

Network:
  - None (SQLite local file)

Scalability:
  - Works with any number of workers
  - Works with any number of jobs
  - Linear with number of orphans detected

================================================================================
MONITORING & ALERTING
================================================================================

Key Metrics to Monitor:
  1. ground_truth_orphaned_jobs_detected_total
     - Should be 0 in healthy systems
     - > 0 indicates worker crashes or timeouts
     
  2. ground_truth_orphaned_jobs_requeued_total
     - Should equal orphaned_jobs_detected
     - Mismatch indicates requeue failures

Alert Suggestions:
  - orphaned_jobs_detected > 5 in 1 hour → WARNING
  - orphaned_jobs_detected > 10 in 1 hour → CRITICAL
  - orphaned_jobs_detected != orphaned_jobs_requeued → CRITICAL

Log Alerts:
  - grep -i "detected.*orphaned" logs/ → find detection events
  - grep -i "requeued orphaned" logs/ → find requeue events

================================================================================
TROUBLESHOOTING
================================================================================

Problem: Jobs not being requeued
Solution:
  1. Check logs for "Orphan watchdog started"
  2. Verify timeout setting: echo $DEEP_THINK_ORPHAN_TIMEOUT_MINUTES
  3. Test manually: python3 -c "from store import detect_orphaned_jobs; print(detect_orphaned_jobs())"

Problem: Legitimate slow jobs being requeued
Solution:
  1. Increase timeout: export DEEP_THINK_ORPHAN_TIMEOUT_MINUTES=10
  2. Or: Implement heartbeats (future enhancement)

Problem: High orphan rate detected
Solution:
  1. Check worker logs for crashes/errors
  2. Monitor worker resource usage (memory, CPU, disk)
  3. Consider scaling workers up
  4. Check database connectivity
